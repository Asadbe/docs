The Users tab manages application users for the generated project. It shows user capacity, existing users, user type, role, login, email, phone, and actions.

</Frame>

## Invite users

Click **Invite User** to add a client, employee, admin, or other application user to the generated system.

<Frame>
  <img src="/images/ugen-dashboard-users-invite.png" alt="Ucode invite user modal" />
</Frame>

The invite modal lets you choose:

- **User type** such as Admin
- **Role** such as Admin
- **Invite method**: Login, Phone, Email, or Link
- **Login** and **Password** when adding a direct credential-based user

This is useful when a client wants to add employees to their own system. For example, a restaurant admin panel can invite managers, operators, or branch employees with login/password access, or share an invite link depending on the preferred onboarding flow.

## What you can do

- View current user usage, such as `1 / 50`
- Invite a new user
- Review user type and role
- See login, email, and phone fields
- Use table pagination for larger user lists
- Upgrade the plan when the user limit is reached

## Invite methods

| Method | When to use it |
| --- | --- |
| Login | Create a user with direct login and password. Good for employees who need immediate access to an internal system. |
| Phone | Invite or identify users by phone number when the product uses SMS, call center, or phone-based workflows. |
| Email | Invite users through email when the system should send onboarding or account messages. |
| Link | Share an invite link when the client wants users to join themselves without manually creating every account. |

## Columns

| Column | Purpose |
| --- | --- |
| Name | User display name |
| User Type | Application user type |
| Role | Assigned role |
| Login | Login identifier |
| Email | User email |
| Phone | Optional phone number |
| Action | User management actions |

## Recommended setup

Create clear user types before inviting many people. For example, a client system can have Admin, Manager, Operator, and Viewer users. Then use Permissions to decide what each user type can see and do.
