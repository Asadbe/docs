The Logs tab shows system activity, API requests, and serverless function execution history. It is the first place to check when data is not loading, an API call returns an unexpected result, or a generated workflow needs debugging.

<Frame>

</Frame>

## Filters

Logs include controls for:

- Log type
- Action
- Time range
- Collection

Use filters when the project has many requests. For example, filter by collection to inspect only `research_stations`, or filter by action to review only list requests, create requests, updates, or failures.

## Example events

The project log can show API activity such as:

- `GET /api/v2/items/client_type`
- `GET /api/v2/items/role`
- `GET /api/v2/items/alerts`
- `GET /api/v2/items/weather_records`
- `GET /api/v2/items/research_stations`
- `GET /api/v2/items/ice_coverage_data`

Use Logs to debug data loading, API calls, integrations, and server function behavior.

## Open a log

<Frame>
  <img src="/images/ugen-dashboard-logs-detail.png" alt="Ucode opened log with request details, status, timing, and collection" />
</Frame>

Open a log row to inspect the exact request. The detail view shows:

- **URL** for the endpoint that was called
- **Method**, such as `GET`
- **Status**, such as `204`
- **Request duration** in milliseconds
- **Started** and **Finished** timestamps
- **Action by**, for example `system`
- **Action type**, such as `GET_LIST`
- **Collection**, such as `research_stations`
- **Request** and **Response** tabs for deeper debugging

This helps the team understand whether the issue is in the frontend, API, collection permissions, database data, or server function logic.

## Debugging flow

1. Reproduce the issue in Preview or the live app.
2. Open Logs and filter by the affected collection or action.
3. Open the newest matching log.
4. Check method, URL, status, duration, and response.
5. If the request is blocked, review Permissions and API keys.
6. If the request succeeds but data is wrong, review Database records and schema.
