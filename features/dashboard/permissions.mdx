Permissions define what each user type can access inside the generated application. Use this page after the permission data finishes loading, then configure access for the selected role.

## Permission areas

The Permissions tab includes:

- **User type selector** such as Admin
- **Table Permissions** for database-level access
- **Custom Permissions** for project-specific rules
- **Save Changes** action
- Granular permission loading for detailed rules

## Table permission actions

| Permission | Meaning |
| --- | --- |
| Read | Allows the role to view records from the table. |
| Write | Allows the role to create new records. |
| Update | Allows the role to edit existing records. |
| Delete | Allows the role to remove records. |
| Public | Allows broader access when a table is intended to be visible without strict user restrictions. |

## How permissions work

Permissions are usually controlled per user type. For each role, you can decide which parts of the generated app should be available and which database resources the role can read, create, update, or delete.

Common examples:

- Disable admin-only menu pages for regular employees.
- Allow a manager to view reports but prevent database schema changes.
- Give operators access to orders and customers but hide settings.
- Allow a support user to read user records but not delete them.
- Restrict sensitive tables such as payments, API keys, or internal reports.

## Recommended workflow

1. Select the user type.
2. Wait until granular permissions finish loading.
3. Review table permissions.
4. Disable menu pages or table actions the role should not access.
5. Add custom permissions for special workflows.
6. Save changes.
7. Test the app as a user with that role.

<Warning>
  Be careful with Delete and Public access. These permissions can expose or remove important project data if they are enabled for the wrong role.
</Warning>
